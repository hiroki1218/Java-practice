# Java基礎：型・変数・メモリ
 
## 1. 拡大変換と縮小変換
 
### 拡大変換（Widening）
- **小さい型 → 大きい型**への変換
- 情報が失われないため、**自動で変換される**（明示的キャスト不要）
 
```java
int num = 100;
long bigNum = num; // int → long 自動変換
```
 
### 縮小変換（Narrowing）
- **大きい型 → 小さい型**への変換
- 情報が失われる可能性があるため、**明示的なキャストが必要**
 
```java
long bigNum = 100L;
int num = (int) bigNum; // long → int キャストが必要
```
 
### なぜ縮小変換は危険なのか？
- 各型にはメモリサイズがある（int: 32bit、long: 64bit）
- 大きい箱の中身を小さい箱に押し込むと、溢れてデータが壊れる
 
```java
long big = 3000000000L; // 30億（intの最大値 約21億を超えている）
int small = (int) big;
System.out.println(small); // -1294967296 ← 意図しない値になる
```

## 2. 演算における型のルール
 
### 大原則
**演算は、その式の中で最も大きい型に合わせて行われる。**
 
### ルール一覧
| ルール | 説明 |
|--------|------|
| 同じ型同士の演算 | その型で結果が返る（int同士ならint） |
| 異なる型同士の演算 | 大きい方に揃えてから計算される |
| byte/short/charの演算 | 自動的にintに格上げされる |
 
---
 
## 3. int同士の割り算の注意点
 
```java
int a = 7;
int b = 2;
System.out.println(a / b); // 3（3.5ではない）
```
 
- int同士の演算結果はintになるため、小数部分が切り捨てられる
 
### 片方がdoubleなら結果もdouble
 
```java
int a = 7;
double b = 2.0;
System.out.println(a / b); // 3.5
```
 
- intがdoubleに拡大変換されてから、double同士の演算になる
 
---
 
## 4. よくある落とし穴
 
```java
double result = 7 / 2;
System.out.println(result); // 3.0（3.5ではない！）
```
 
### なぜ3.5にならないのか？
1. **まず右辺の式が評価される** → int同士の計算なので結果は `3`
2. **その後、左辺の変数に代入される** → `3` がdoubleに変換されて `3.0`
 
右辺の計算と左辺への代入は完全に独立している。左辺の型は右辺の計算に影響しない。
 
### 正しい書き方
 
```java
double result = 7.0 / 2;        // OK: 3.5
double result2 = (double) 7 / 2; // OK: 3.5（明示的キャスト）
```

## 5. byte/short/charの演算は自動的にintになる
 
```java
byte a = 10;
byte b = 20;
byte c = a + b; // コンパイルエラー！
```
 
- JVMの内部処理が基本的にint単位で行われるため、自動的にintに格上げされる
 
```java
byte a = 10;
byte b = 20;
int c = a + b;           // OK
byte d = (byte)(a + b);  // 明示的キャストすればbyteにも入れられる
```

## 6. 変数のスコープ（`{}` との関係）
 
### スコープとは
変数が**使える範囲**のこと。Javaでは `{}` がスコープの境界になる。
 
### 基本ルール
- 変数は**宣言された `{}` の中でだけ有効**
- `{}` を抜けた瞬間、その変数は消滅する
 
```java
public void example() {
    int a = 10; // メソッド全体で使える
 
    if (a > 5) {
        int b = 20; // このif文の {} の中でだけ使える
        System.out.println(a); // OK: aは外側のスコープ
        System.out.println(b); // OK: bはこのスコープ内
    }
 
    System.out.println(a); // OK
    System.out.println(b); // コンパイルエラー！ bはifの{}を出たので消滅済み
}
```
 
### なぜスコープがあるのか？
- 不要になった変数のメモリを早く解放できる
- 変数名の衝突を防げる
- 「この変数はここでしか使わない」という意図を明示できる
 
### スコープの入れ子
内側のスコープから外側の変数は見える。逆は見えない。
 
```java
{
    int x = 1;
    {
        int y = 2;
        System.out.println(x); // OK: 外側の変数は見える
        System.out.println(y); // OK
    }
    System.out.println(x); // OK
    System.out.println(y); // エラー: 内側の変数は見えない
}
```

## 7. finalが禁止していること（矢印の付け替え）
 
### finalとは
変数に `final` をつけると、**再代入が禁止される**。
 
### 「矢印の付け替え」で理解する
変数は「値そのもの」ではなく、**値を指し示す矢印**と考える。
 
```java
final int num = 10;
num = 20; // コンパイルエラー！ 矢印の付け替え禁止
```
 
finalは「矢印を一度セットしたら、別の場所に向け直すことはできない」という制約。
 
### プリミティブ型の場合
矢印の先が値そのものなので、**値の変更が不可能**。
 
```java
final int age = 25;
age = 30; // エラー: 再代入できない
```
 
### 参照型の場合（ここが重要）
矢印の先は**オブジェクトのアドレス**。finalはアドレスの付け替えを禁止するだけで、**オブジェクトの中身は変更できる**。
 
```java
final List<String> names = new ArrayList<>();
names.add("Hiroki");  // OK: 中身の変更は許される
names.add("Taro");    // OK: 中身の変更は許される
names = new ArrayList<>(); // エラー！ 矢印の付け替えは禁止
```
 
### なぜこの区別が重要なのか？
「finalをつけたから安全」と思い込むと、参照型の中身が書き換えられることを見落とす。finalが守るのは**矢印の向き先だけ**で、**矢印の先にあるもの自体は守らない**。

## 8. `new` とメモリのライフサイクル（生成 → 参照 → GC回収）
 
### 3つのフェーズ
 
#### フェーズ1: 生成（new）
`new` はヒープメモリ上に**オブジェクトの実体を作る**命令。
 
```java
String name = new String("Hiroki");
```
 
このとき何が起きているか：
1. ヒープメモリにStringオブジェクトが確保される
2. 変数 `name` はスタックメモリ上に作られる
3. `name` にはオブジェクトの**アドレス（参照）**が入る
 
#### フェーズ2: 参照
変数を通じてオブジェクトにアクセスしている状態。
 
```java
System.out.println(name); // nameの矢印をたどってオブジェクトにアクセス
String another = name;     // 同じオブジェクトを指す矢印がもう1本増える
```
 
1つのオブジェクトを複数の変数が参照できる。この時点では、どれか1つでも矢印が残っていればオブジェクトは生きている。
 
#### フェーズ3: GC回収（ガベージコレクション）
**どの変数からも参照されなくなったオブジェクト**は、GCが自動的にメモリから回収する。
 
```java
String name = new String("Hiroki"); // オブジェクト生成、nameが参照
name = null;                         // 矢印を切る → 誰も参照していない
// → GCがいずれ回収する
```
 
### GCの重要なポイント
- **いつ回収されるかは保証されない**（JVMが適切なタイミングで実行する）
- プログラマが明示的にメモリを解放する必要がない（C言語のfreeとの違い）
- `System.gc()` で「お願い」はできるが、実行は保証されない
 
### スコープとGCの関係
スコープを抜けると変数が消える → 参照が切れる → GCの対象になる。
 
```java
public void example() {
    String name = new String("Hiroki"); // オブジェクト生成
    System.out.println(name);
} // メソッド終了 → name消滅 → オブジェクトへの参照がなくなる → GC対象
```
 
セクション6のスコープ、セクション7のfinal、セクション8のメモリライフサイクルは**すべて繋がっている**。スコープが変数の寿命を決め、finalが参照の付け替えを制限し、参照がなくなったオブジェクトをGCが回収する。
